## Getting Started

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="what-is-duck-iam">
    <AccordionTrigger>What is duck-iam?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      duck-iam is a TypeScript authorization engine that combines Role-Based Access Control (RBAC) and
      Attribute-Based Access Control (ABAC) into a single evaluation pipeline. You define roles with
      permissions, write policies with conditions, and the engine evaluates them together to produce
      an allow or deny decision. Everything is type-safe -- actions, resources, scopes, and role IDs
      are checked at compile time when using `createAccessConfig()`.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="rbac-vs-abac">
    <AccordionTrigger>How does duck-iam compare to pure RBAC or pure ABAC?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Pure RBAC is simple but rigid -- you cannot express rules like "editors can only update their own
      posts." Pure ABAC is powerful but verbose -- every permission requires a full policy rule, even
      for basic role checks. duck-iam combines both: roles handle common patterns (viewer, editor, admin)
      and policies handle fine-grained rules (owner-only access, time-based restrictions, IP filtering).
      Roles are internally converted to an ABAC policy via `rolesToPolicy()`, so the same condition
      engine handles everything in one unified pipeline.
    </AccordionContent>
  </AccordionItem>

  </AccordionItem>

## Roles and Permissions

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="what-is-role">
    <AccordionTrigger>What is a role in duck-iam?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      A role is a named collection of permissions. Each permission is an action-resource pair like
      `read:post` or `delete:comment`. Roles can inherit from other roles to form hierarchies.
      Subjects (users) are assigned one or more roles, and the engine resolves all permissions
      transitively through the inheritance chain.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="role-inheritance">
    <AccordionTrigger>How does role inheritance work?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Roles can inherit from other roles using the `inherits` property in `defineRole().inherits('parent')`.
      When a subject has the `editor` role, and `editor` inherits from `viewer`, the subject gets all
      of `viewer`'s permissions automatically. Inheritance chains are resolved transitively -- if `admin`
      inherits `editor`, which inherits `viewer`, then `admin` has all three sets of permissions.
      The engine uses a visited-set to prevent infinite loops from circular inheritance.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="multiple-inheritance">
    <AccordionTrigger>Can a role inherit from multiple parents?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. A role can inherit from as many parents as you need. For example, a `moderator` role can
      inherit from both `viewer` and `commenter`:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const moderator = defineRole('moderator')
  .inherits('viewer', 'commenter')
  .grant('delete', 'comment')
  .build()`}</code></pre>
      <p className="mt-3">
        The engine collects permissions from all ancestors. Duplicate permissions are deduplicated
        automatically.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="circular-inheritance">
    <AccordionTrigger>What happens with circular role inheritance?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The engine handles it safely. During role resolution, it maintains a visited-set of role IDs.
      If a role has already been visited, it is skipped, breaking the cycle. `validateRoles()` will
      report circular inheritance as a warning (not an error) because the engine handles it gracefully
      at runtime. However, circular inheritance is almost always a configuration mistake -- fix the
      cycle when you see the warning.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="multiple-roles-per-user">
    <AccordionTrigger>Can a subject have multiple roles?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. A subject can have any number of roles assigned. The engine resolves permissions from all
      assigned roles (including inherited ones) and combines them into a single synthetic `__rbac__`
      policy with `allow-overrides`. This means if any role grants a permission, the RBAC layer allows
      it. Policies can then add further restrictions on top of that.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="roles-to-policy">
    <AccordionTrigger>How are roles converted to policies internally?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The engine calls `rolesToPolicy()` which takes the subject's resolved roles and creates a synthetic
      policy with ID `__rbac__`. Each permission becomes a rule with `effect: 'allow'` and the matching
      action and resource. This policy uses `allow-overrides` as its combining algorithm -- if any
      permission matches, the RBAC layer allows the request. This policy is then evaluated alongside
      your custom ABAC policies.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="wildcard-permissions">
    <AccordionTrigger>Can I use wildcards in role permissions?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. Use `'*'` to match any action or resource:
      <ul className="mt-2 list-disc pl-5">
        <li>`grant('*', 'post')` -- all actions on posts</li>
        <li>`grant('read', '*')` -- read access on all resources</li>
        <li>`grant('*', '*')` -- full access to everything (superadmin)</li>
      </ul>
      <p className="mt-3">
        Wildcards are matched during rule evaluation. A rule with `actions: ['*']` matches any action
        in the request.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="role-scoped-permissions">
    <AccordionTrigger>Can a role's permissions be limited to a specific scope?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. Permissions in a role can include an optional `scope` field. When present, that permission
      only matches if the request scope matches. For example:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const orgAdmin = defineRole('org-admin')
  .grant('manage', 'user', 'org-1')
  .build()`}</code></pre>
      <p className="mt-3">
        This grants `manage:user` only when the request scope is `org-1`. Requests with a different
        scope or no scope will not match this permission.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="validate-roles">
    <AccordionTrigger>How do I validate my role configuration?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Call `validateRoles()` at application startup:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`import { validateRoles } from '@gentleduck/iam'

const result = validateRoles([viewer, editor, admin])
if (!result.valid) {
  throw new Error(result.issues.map(i => i.message).join(', '))
}`}</code></pre>
      <p className="mt-3">
        It checks for: duplicate role IDs (error), dangling inherits referencing missing roles (error),
        circular inheritance (warning), and empty roles with no permissions and no inheritance (warning).
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="hierarchical-resources">
    <AccordionTrigger>What are hierarchical resource types?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Resource types can use dot notation to form hierarchies. A type of `dashboard.users` matches
      rules that target `dashboard` or `dashboard.users`. This means granting `read` on `dashboard`
      also grants `read` on `dashboard.users`, `dashboard.settings`, etc. The engine checks if the
      request resource type starts with the rule's resource type followed by a dot.
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Policies and Rules

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="what-is-policy">
    <AccordionTrigger>What is a policy?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      A policy is a named container of rules with a combining algorithm. Each rule defines an
      effect (allow or deny), a set of actions and resources it applies to, a priority, and optional
      conditions. The combining algorithm determines how multiple matching rules produce a single
      policy-level result. Policies represent the ABAC side of duck-iam -- they can express any
      authorization logic that roles alone cannot handle.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="when-use-policy">
    <AccordionTrigger>When should I use a policy instead of just roles?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Use policies when you need logic that depends on attributes or context:
      <ul className="mt-2 list-disc pl-5">
        <li>Owner-only access ("users can only edit their own posts")</li>
        <li>Time-based restrictions ("no deployments on weekends")</li>
        <li>IP or location filtering ("admin panel only from office network")</li>
        <li>Status-based rules ("cannot edit locked or archived resources")</li>
        <li>Conditional denials ("deny delete unless user is admin")</li>
      </ul>
      <p className="mt-3">
        If the decision only depends on "does this user have role X" then roles are sufficient.
        If it depends on "who owns this resource" or "what time is it" then you need a policy.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="combining-algorithms">
    <AccordionTrigger>What combining algorithms are available?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Each policy has a combining algorithm that determines how its rules produce a single result:
      <ul className="mt-2 list-disc pl-5">
        <li><strong>deny-overrides</strong> -- if any matching rule denies, the policy denies. Safest default for security-sensitive policies.</li>
        <li><strong>allow-overrides</strong> -- if any matching rule allows, the policy allows. Used internally for the RBAC policy.</li>
        <li><strong>first-match</strong> -- the first matching rule (by insertion order) wins. Good for ordered rule lists.</li>
        <li><strong>highest-priority</strong> -- the matching rule with the highest priority number wins. Good for layered overrides.</li>
      </ul>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="cross-policy-combination">
    <AccordionTrigger>How are multiple policies combined together?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Across policies, duck-iam uses AND-combination: all policies must allow for the overall result
      to be allow. A deny from any single policy means the overall result is deny. This gives you
      defense-in-depth -- each policy is an independent security gate. The RBAC policy (from roles)
      is one of these policies, so roles must grant the permission AND all custom policies must allow it.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="default-effect">
    <AccordionTrigger>What happens when no rules match in a policy?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The engine uses a `defaultEffect` which defaults to `'deny'`. If no rules match in a policy,
      the policy returns the default effect. Since the default is deny, this means "if nothing
      explicitly allows it, it is denied." You can change the default effect per-engine by passing
      `defaultEffect: 'allow'` in the engine options, but this is not recommended for production.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="policy-targets">
    <AccordionTrigger>What are policy targets?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Targets are optional filters on a policy that determine whether the policy is even evaluated
      for a given request. A policy can target specific actions, resources, or roles:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const restrictedPolicy = policy('restricted')
  .target({ actions: ['delete'], resources: ['post'], roles: ['editor'] })
  .algorithm('deny-overrides')
  .rule(...)
  .build()`}</code></pre>
      <p className="mt-3">
        This policy is only evaluated when the action is `delete`, the resource is `post`, and the
        subject has the `editor` role. For all other requests, the policy's rules are skipped. In
        explain output, check `targetMatch` to confirm that no rule evaluation happened.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="rule-priority">
    <AccordionTrigger>What does rule priority do?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Priority is a number on each rule. Its meaning depends on the combining algorithm:
      <ul className="mt-2 list-disc pl-5">
        <li><strong>highest-priority</strong> -- the matching rule with the highest priority number wins. This is where priority matters most.</li>
        <li><strong>deny-overrides / allow-overrides</strong> -- priority is ignored; the algorithm only cares about deny or allow.</li>
        <li><strong>first-match</strong> -- priority is ignored; insertion order determines which rule wins.</li>
      </ul>
      <p className="mt-3">
        Higher numbers mean higher priority. A rule with priority 100 beats a rule with priority 10.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="rule-matching">
    <AccordionTrigger>How does the engine decide if a rule matches?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      A rule matches when all three checks pass:
      <ol className="mt-2 list-decimal pl-5">
        <li><strong>Action match</strong> -- the request action is in the rule's `actions` array, or the rule has `['*']`.</li>
        <li><strong>Resource match</strong> -- the request resource type is in the rule's `resources` array, or the rule has `['*']`, or a hierarchical match (e.g., rule targets `dashboard` and request is `dashboard.users`).</li>
        <li><strong>Conditions met</strong> -- all condition groups evaluate to true (or the rule has no conditions).</li>
      </ol>
      <p className="mt-3">
        All three must pass. If the action matches but the resource does not, the rule is skipped.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="validate-policy">
    <AccordionTrigger>How do I validate a policy from an external source?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Use `validatePolicy()` before saving or using untrusted policy data:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`import { validatePolicy } from '@gentleduck/iam'

const result = validatePolicy(policyFromAPI)
if (!result.valid) {
  console.error(result.issues.map(i => i.message).join(', '))
}`}</code></pre>
      <p className="mt-3">
        It checks for: required fields (`id`, `name`, `algorithm`, `rules`), valid combining algorithm,
        valid effect values (`allow` / `deny`), valid condition operators, correct condition group
        structure, duplicate rule IDs, and valid target structure.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="standalone-rules">
    <AccordionTrigger>Can I define rules outside of a policy and reuse them?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. Use `defineRule()` to create standalone rules:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const ownerRule = defineRule('owner-check')
  .allow()
  .on('update', 'delete')
  .of('post')
  .priority(10)
  .when(w => w.isOwner())
  .build()

// Add to any policy:
const p = policy('my-policy')
  .algorithm('deny-overrides')
  .addRule(ownerRule)
  .build()`}</code></pre>
      <p className="mt-3">
        This is useful when the same rule logic applies across multiple policies.
      </p>
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Conditions and Operators

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="how-conditions-work">
    <AccordionTrigger>How do conditions work?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Conditions are expressions attached to rules that must evaluate to true for the rule to match.
      Each condition compares a field value from the request context against an expected value using
      an operator. Fields are dot-notation paths like `resource.attributes.ownerId` or
      `environment.time`. Conditions are grouped into logical blocks (`all`, `any`, `none`) that
      can be nested.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="all-operators">
    <AccordionTrigger>What operators are available for conditions?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      <ul className="mt-2 list-disc pl-5">
        <li><strong>eq</strong> -- equals (strict equality)</li>
        <li><strong>neq</strong> -- not equals</li>
        <li><strong>gt</strong> -- greater than</li>
        <li><strong>gte</strong> -- greater than or equal</li>
        <li><strong>lt</strong> -- less than</li>
        <li><strong>lte</strong> -- less than or equal</li>
        <li><strong>in</strong> -- value is in an array</li>
        <li><strong>nin</strong> -- value is not in an array</li>
        <li><strong>contains</strong> -- array contains the value</li>
        <li><strong>not_contains</strong> -- array does not contain the value</li>
        <li><strong>starts_with</strong> -- string starts with the value</li>
        <li><strong>ends_with</strong> -- string ends with the value</li>
        <li><strong>matches</strong> -- string matches a regex pattern</li>
        <li><strong>exists</strong> -- field is not null/undefined</li>
        <li><strong>not_exists</strong> -- field is null/undefined</li>
        <li><strong>subset_of</strong> -- array is a subset of the value array</li>
        <li><strong>superset_of</strong> -- array is a superset of the value array</li>
      </ul>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="condition-groups">
    <AccordionTrigger>What are condition groups (all, any, none)?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Conditions are organized into logical groups:
      <ul className="mt-2 list-disc pl-5">
        <li><strong>all</strong> -- AND logic. Every condition in the group must pass.</li>
        <li><strong>any</strong> -- OR logic. At least one condition must pass.</li>
        <li><strong>none</strong> -- NOT logic. No condition may pass (all must fail).</li>
      </ul>
      <p className="mt-3">
        Groups can be nested up to 10 levels deep. For example, you can have an `all` group containing
        an `any` group and a `none` group to express: "must be (admin OR owner) AND must NOT be banned."
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="dollar-variables">
    <AccordionTrigger>What are $-variables in condition values?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Values starting with `$` are resolved dynamically at evaluation time from the request context:
      <ul className="mt-2 list-disc pl-5">
        <li><code className="rounded bg-muted px-2 py-1">$subject.id</code> -- the requesting subject's ID</li>
        <li><code className="rounded bg-muted px-2 py-1">$subject.roles</code> -- the subject's resolved roles array</li>
        <li><code className="rounded bg-muted px-2 py-1">$subject.attributes.department</code> -- a subject attribute</li>
        <li><code className="rounded bg-muted px-2 py-1">$resource.id</code> -- the resource ID</li>
        <li><code className="rounded bg-muted px-2 py-1">$resource.type</code> -- the resource type</li>
        <li><code className="rounded bg-muted px-2 py-1">$resource.attributes.ownerId</code> -- a resource attribute</li>
        <li><code className="rounded bg-muted px-2 py-1">$environment.time</code> -- an environment value</li>
      </ul>
      <p className="mt-3">
        Without the `$` prefix, the value is treated as a static literal. With it, the engine
        resolves the actual value from the request context at runtime.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="field-resolution">
    <AccordionTrigger>How does field resolution work?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Fields in conditions are dot-notation paths resolved against the AccessRequest object. The engine
      splits the path by dots and walks the object tree. Available root paths:
      <ul className="mt-2 list-disc pl-5">
        <li><code className="rounded bg-muted px-2 py-1">subject.id</code>, <code className="rounded bg-muted px-2 py-1">subject.roles</code>, <code className="rounded bg-muted px-2 py-1">{"subject.attributes.*"}</code></li>
        <li><code className="rounded bg-muted px-2 py-1">resource.type</code>, <code className="rounded bg-muted px-2 py-1">resource.id</code>, <code className="rounded bg-muted px-2 py-1">{"resource.attributes.*"}</code></li>
        <li><code className="rounded bg-muted px-2 py-1">{"environment.*"}</code></li>
      </ul>
      <p className="mt-3">
        Unresolved paths return `undefined`. `__proto__`, `constructor`, and `prototype` are blocked
        in all paths to prevent prototype pollution.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="isowner-shortcut">
    <AccordionTrigger>What does the isOwner() condition shortcut do?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Expands to:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`{ field: 'resource.attributes.ownerId', operator: 'eq', value: '$subject.id' }`}</code></pre>
      <p className="mt-3">
        Requires `ownerId` in resource attributes when calling `engine.can()` or `engine.check()`.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="why-dollar-prefix">
    <AccordionTrigger>Why does duck-iam use the $ prefix for dynamic value references?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Without a marker, `'subject.id'` in a condition value is compared as a literal string, not a runtime
      path. The `$` prefix tells the engine to strip it and resolve the remaining path against the
      request context.
      <p className="mt-3">
        Alternative designs (separate `rightField` property, `{"{"} $ref: 'subject.id' {"}"}`) would add
        type complexity and schema changes. `$` is one `if` in `resolveValue()` and follows a convention
        already understood from shell variables, MongoDB operators, and template strings.
      </p>
      <p className="mt-3">
        With <code className="rounded bg-muted px-2 py-1">createAccessConfig()</code> and a typed context,
        <code className="rounded bg-muted px-2 py-1">DollarPaths&lt;TContext&gt;</code> generates valid
        <code className="rounded bg-muted px-2 py-1">$</code>-prefixed paths for IDE autocomplete. No <code className="rounded bg-muted px-2 py-1">as string</code> cast needed.
        Literal values starting with `$` are not supported, but this never arises in practice.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="dollar-resolution-walkthrough">
    <AccordionTrigger>How does $ variable resolution work step-by-step?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      When a condition is evaluated, the engine processes both sides of the comparison:
      <ol className="mt-2 list-decimal pl-5">
        <li><strong>Left side (field)</strong> -- always resolved as a dot-path against the request. For example,
          `'resource.attributes.ownerId'` walks `request.resource.attributes.ownerId`.</li>
        <li><strong>Right side (value)</strong> -- if it starts with `$`, the engine strips the `$` and resolves
          it as a dot-path too. Otherwise it is used as-is.</li>
      </ol>
      <p className="mt-3">
        For example, `isOwner()` produces this condition:
      </p>
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`{ field: 'resource.attributes.ownerId', operator: 'eq', value: '$subject.id' }`}</code></pre>
      <p className="mt-3">
        Given a request where Alice deletes her own post:
      </p>
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`subject: { id: 'user-alice' }
resource: { attributes: { ownerId: 'user-alice' } }`}</code></pre>
      <ol className="mt-2 list-decimal pl-5">
        <li>`resolve('resource.attributes.ownerId')` returns `'user-alice'`</li>
        <li>`resolveValue('$subject.id')` strips `$`, then `resolve('subject.id')` returns `'user-alice'`</li>
        <li>`eq('user-alice', 'user-alice')` returns `true` -- allowed</li>
      </ol>
      <p className="mt-3">
        If Bob tries the same request, `resolve('subject.id')` returns `'user-bob'`, and
        `eq('user-alice', 'user-bob')` returns `false` -- denied.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="role-condition">
    <AccordionTrigger>Can I check roles inside a condition?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. Use the `role()` shortcut on the condition builder:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`when(w => w.role('admin'))`}</code></pre>
      <p className="mt-3">
        This expands to `{"{"} field: 'subject.roles', operator: 'contains', value: 'admin' {"}"}`.
        You can also write it manually with `.check('subject.roles', 'contains', 'admin')`.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="not-conditions">
    <AccordionTrigger>How do I negate conditions?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Use the `not()` method on the condition builder or a `none` group:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`// Using not() in builder:
.when(w => w
  .not(w => w.role('banned'))
  .isOwner()
)

// Raw condition group:
{ none: [{ field: 'subject.roles', operator: 'contains', value: 'banned' }] }`}</code></pre>
      <p className="mt-3">`none` is logical NOT -- the group passes when all inner conditions fail.</p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="regex-conditions">
    <AccordionTrigger>Can I use regex in conditions?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. Use the `matches` operator:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`.check('resource.attributes.email', 'matches', '^.*@company\\.com$')`}</code></pre>
      <p className="mt-3">The value is a regex pattern string. Avoid complex patterns to prevent ReDoS.</p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="condition-undefined">
    <AccordionTrigger>What happens when a condition field resolves to undefined?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Most operators return `false` when the field resolves to `undefined`. Use `exists` (non-null/non-undefined)
      and `not_exists` (null/undefined) to handle optional fields explicitly.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="nesting-limit">
    <AccordionTrigger>Is there a limit on condition nesting depth?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. Condition groups can be nested up to 10 levels deep. The engine enforces this limit
      during evaluation to prevent stack overflows from excessively deep or malicious condition
      trees. If you hit this limit, simplify your condition structure by flattening nested groups
      or splitting logic across multiple rules or policies.
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Multi-Tenant and Scoping

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="scoped-roles">
    <AccordionTrigger>What are scoped roles?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Scoped roles allow a subject to have different roles in different contexts (tenants, organizations,
      workspaces). For example, a user can be `admin` in `org-1` and `viewer` in `org-2`. When you pass
      a `scope` parameter to `engine.can()`, the engine calls `adapter.getSubjectScopedRoles()` to get
      the extra roles for that scope and merges them with the subject's base roles before evaluation.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="scoping-levels">
    <AccordionTrigger>What are the different levels of scoping?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      duck-iam supports scoping at three levels:
      <ul className="mt-2 list-disc pl-5">
        <li><strong>Assignment-level scoping</strong> -- A subject has different roles in different scopes. This is the most common pattern for multi-tenant apps. Handled by `getSubjectScopedRoles()` in the adapter.</li>
        <li><strong>Permission-level scoping</strong> -- Individual permissions within a role are limited to a scope. Set via `grant('action', 'resource', 'org-1')` or `grantScoped('org-1', 'action', 'resource')` in the role builder.</li>
        <li><strong>Role-level scoping</strong> -- The entire role definition is constrained to a scope. Set via `defineRole('admin').scope('org-1')`.</li>
      </ul>
      <p className="mt-3">
        You can combine these levels. Most applications use assignment-level scoping as it is the
        simplest and most flexible.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="pass-scope">
    <AccordionTrigger>How do I pass a scope when checking permissions?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Pass the scope as the fifth parameter to `engine.can()` or `engine.check()`:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`await engine.can('user-1', 'manage', { type: 'dashboard', attributes: {} }, undefined, 'org-1')
// 4th param is environment (or undefined), 5th is scope`}</code></pre>
      <p className="mt-3">
        In server middleware, the scope is typically extracted from the request (URL path, header,
        or JWT claim) and passed automatically.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="no-scope-adapter">
    <AccordionTrigger>What if my adapter does not support scoped roles?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      If your adapter does not implement `getSubjectScopedRoles()`, the engine falls back to an empty
      array -- no extra scoped roles are added. The subject's base roles are still used. You can always
      add scoping support later by implementing that method on your adapter.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="scope-in-conditions">
    <AccordionTrigger>Can I reference the scope in policy conditions?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. The scope is available in the AccessRequest context. You can write conditions that
      check the scope value alongside other attributes. For example, you could create a condition
      that only allows access when the resource's `orgId` matches the request scope, adding an
      extra layer of tenant isolation on top of scoped role assignments.
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Engine and Evaluation

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="engine-constructor">
    <AccordionTrigger>What options can I pass to the Engine constructor?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`new Engine({
  adapter,              // Required: the data adapter
  cacheTTL: 60,         // Cache time-to-live in seconds (default: 60)
  maxCacheSize: 1000,         // Max entries in the subject cache (default: 1000)
  defaultEffect: 'deny',     // Default when no rules match (default: 'deny')
  hooks: { ... },       // Optional lifecycle hooks
})`}</code></pre>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="can-vs-check">
    <AccordionTrigger>What is the difference between engine.can() and engine.check()?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      `can()` returns a **boolean** -- `true` if allowed, `false` if denied. `check()` returns
      the full **Decision object** with `allowed`, `effect`, `reason`, `policy`, `rule`, and timing
      information. Use `can()` for simple guards and `check()` when you need to log or explain the decision:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`// Simple boolean:
const allowed = await engine.can('user-1', 'read', { type: 'post', attributes: {} })

// Full decision with details:
const decision = await engine.check('user-1', 'read', { type: 'post', attributes: {} })
// decision.allowed, decision.reason, decision.policy, decision.rule`}</code></pre>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="decision-object">
    <AccordionTrigger>What does the Decision object contain?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`interface Decision {
  allowed: boolean
  effect: 'allow' | 'deny'
  rule?: Rule
  policy?: string
  reason: string
  duration: number
  timestamp: number
}`}</code></pre>
      <p className="mt-3">
        Check `result.allowed` for the boolean. Use `result.reason`, `result.policy`, and
        `result.rule?.id` when you need to log or explain why a decision was produced.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="evaluation-order">
    <AccordionTrigger>What is the evaluation order?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The engine follows these steps in order:
      <ol className="mt-2 list-decimal pl-5">
        <li><strong>Load subject</strong> -- get roles and attributes from the adapter (cached)</li>
        <li><strong>Resolve scoped roles</strong> -- if a scope is passed, merge scoped roles with base roles</li>
        <li><strong>Run beforeEvaluate hook</strong> -- allows request enrichment</li>
        <li><strong>Load policies</strong> -- load the cached synthetic `__rbac__` policy plus custom ABAC policies</li>
        <li><strong>Evaluate all policies</strong> -- iterate through RBAC + ABAC policies, evaluate rules</li>
        <li><strong>AND-combine results</strong> -- all policies must allow, any deny means overall deny</li>
        <li><strong>Run afterEvaluate hook</strong> -- for audit logging</li>
        <li><strong>Run onDeny hook</strong> -- if the result was denied</li>
        <li><strong>Run onError hook</strong> -- only when an exception occurs; the check then fails closed</li>
      </ol>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="permissions-batch">
    <AccordionTrigger>How do I check multiple permissions at once?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Use `engine.permissions()` for batch checks:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const checks = [
  { action: 'create', resource: 'post' },
  { action: 'update', resource: 'post', resourceId: 'post-1' },
  { action: 'delete', resource: 'post', resourceId: 'post-1' },
  { action: 'manage', resource: 'dashboard' },
]

const perms = await engine.permissions('user-1', checks)
// { 'create:post': true, 'update:post:post-1': false, ... }`}</code></pre>
      <p className="mt-3">
        This is more efficient than calling `can()` multiple times because it loads the subject,
        roles, and policies once and evaluates all checks against the same cached data.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="permission-map-keys">
    <AccordionTrigger>How are permission map keys formatted?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The keys in the permission map returned by `engine.permissions()` are formatted as
      one of these forms:
      <ul className="mt-2 list-disc pl-5">
        <li><code className="rounded bg-muted px-2 py-1">create:post</code> -- action + resource</li>
        <li><code className="rounded bg-muted px-2 py-1">update:post:post-1</code> -- action + resource + resourceId</li>
        <li><code className="rounded bg-muted px-2 py-1">org-1:manage:dashboard</code> -- scope + action + resource</li>
        <li><code className="rounded bg-muted px-2 py-1">org-1:update:post:post-1</code> -- scope + action + resource + resourceId</li>
      </ul>
      <p className="mt-3">
        These keys are generated by the `buildPermissionKey()` utility function. The same format
        is used by client libraries to look up permissions.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="environment-param">
    <AccordionTrigger>What is the environment parameter for?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The environment is an optional object you can pass to `engine.can()` for context that is neither
      subject nor resource. Common examples:
      <ul className="mt-2 list-disc pl-5">
        <li>`environment.time` -- current timestamp for time-based restrictions</li>
        <li>`environment.ip` -- client IP for network-based rules</li>
        <li>`environment.dayOfWeek` -- day of week for schedule-based access</li>
      </ul>
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`await engine.can('user-1', 'deploy', resource, {
  time: Date.now(),
  dayOfWeek: new Date().toLocaleDateString('en', { weekday: 'long' }),
  ip: '192.168.1.1',
})`}</code></pre>
      <p className="mt-3">
        Conditions can reference environment values with paths like `environment.time` or
        `$environment.dayOfWeek`.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="error-handling">
    <AccordionTrigger>What happens when an error occurs during evaluation?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The engine fails closed. If any error occurs during evaluation (adapter failure, condition
      evaluation error, or hook failure), the result defaults to deny. The `onError` hook is called with the
      error for logging. One implementation detail matters: if your `onError` hook itself throws,
      the surrounding call can reject instead of returning a deny object. Keep `onError` defensive.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="dev-vs-prod-mode">
    <AccordionTrigger>What's the difference between development and production mode?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The Engine constructor accepts a `mode` option that controls the evaluation output format and
      available features:
      <ul className="mt-2 list-disc pl-5">
        <li><strong>Development</strong> (`mode: 'development'`) -- returns rich `Decision` objects with timing
          information and reasons, fires lifecycle hooks, and enables `explain()` for debugging.</li>
        <li><strong>Production</strong> (`mode: 'production'`) -- returns plain booleans, skips hooks,
          omits timing data, and is roughly 2x faster than development mode.</li>
      </ul>
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`// Development (default):
const engine = new Engine({ adapter, mode: 'development' })
const decision = await engine.check(subject, 'read', resource)
// decision.allowed, decision.reason, decision.duration

// Production:
const engine = new Engine({ adapter, mode: 'production' })
const allowed = await engine.check(subject, 'read', resource)
// allowed === true | false`}</code></pre>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="prod-mode-bundle-size">
    <AccordionTrigger>Does production mode reduce bundle size?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      No. The `mode` option is a runtime flag -- it does not affect what code is included in your
      bundle. However, if you only need fast boolean checks you can import `evaluateFast` directly
      instead of the full `Engine` class. Because `evaluateFast` is a standalone function with no
      dependency on the Engine internals, bundlers can tree-shake the unused Engine code and produce
      a smaller output.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="explain-in-production">
    <AccordionTrigger>Can I use explain() in production mode?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      No. Calling `explain()` in production mode throws an error because the engine does not collect
      the diagnostic data that `explain()` needs. If you need to debug a specific decision, temporarily
      switch to `mode: 'development'` in that environment or spin up a development-mode instance
      alongside your production one.
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Hooks and Extensibility

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="what-are-hooks">
    <AccordionTrigger>What hooks does the engine support?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Four lifecycle hooks:
      <ul className="mt-2 list-disc pl-5">
        <li><strong>beforeEvaluate</strong> -- runs before evaluation. Can modify the AccessRequest (e.g., enrich with extra attributes from a database). Return the modified request.</li>
        <li><strong>afterEvaluate</strong> -- runs after evaluation with the request and decision. Use for audit logging, metrics, or analytics.</li>
        <li><strong>onDeny</strong> -- runs only when the decision is deny. Use for alerting, rate limiting, or logging denied access attempts.</li>
        <li><strong>onError</strong> -- runs when an error occurs during evaluation. Use for error tracking and monitoring.</li>
      </ul>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="before-evaluate-example">
    <AccordionTrigger>How do I enrich requests with beforeEvaluate?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const engine = new Engine({
  adapter,
  hooks: {
    beforeEvaluate: async (request) => {
      // Fetch the resource from the database to get ownerId
      const post = await db.posts.findUnique({ where: { id: request.resource.id } })
      return {
        ...request,
        resource: {
          ...request.resource,
          attributes: { ...request.resource.attributes, ownerId: post?.authorId },
        },
      }
    },
  },
})`}</code></pre>
      <p className="mt-3">
        This is useful when the caller does not have all the resource attributes at the time of
        the permission check. The hook can fetch them from a database or API.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="hooks-explain">
    <AccordionTrigger>Are hooks called during engine.explain()?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Only `beforeEvaluate` is called during `explain()`. The `afterEvaluate`, `onDeny`, and `onError`
      hooks are not triggered on the explain path. That keeps explain read-only from an audit-hook
      perspective, but a thrown `beforeEvaluate` still rejects the explain call because there is no
      separate explain-side error hook.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="hook-errors">
    <AccordionTrigger>What happens if a hook throws an error?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      If `beforeEvaluate` throws, the evaluation is aborted and the result defaults to deny (fail
      closed). The `onError` hook is called with the error. In the current implementation,
      `afterEvaluate` and `onDeny` also run inside the main try/catch, so they can change an
      otherwise successful check into a deny. If `onError` itself throws, the surrounding call can reject.
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Caching and Performance

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="caching">
    <AccordionTrigger>How does caching work?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The engine maintains four LRU caches:
      <ul className="mt-2 list-disc pl-5">
        <li><strong>Policy cache</strong> -- caches the list of policies from the adapter</li>
        <li><strong>Role cache</strong> -- caches the list of role definitions from the adapter</li>
        <li><strong>RBAC policy cache</strong> -- caches the synthetic `__rbac__` policy built from roles</li>
        <li><strong>Subject cache</strong> -- caches per-subject data (roles, attributes, scoped roles)</li>
      </ul>
      <p className="mt-3">
        All caches have a configurable TTL (default: 60 seconds). The subject cache has a configurable
        max size (default: 1000 entries). Caches are automatically populated on first access and refreshed
        when the TTL expires.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="cache-ttl">
    <AccordionTrigger>How do I change the cache TTL?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Pass `cacheTTL` (in seconds) to the Engine constructor:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const engine = new Engine({ adapter, cacheTTL: 120 })  // 2 minutes
const engine = new Engine({ adapter, cacheTTL: 0 })    // disable caching`}</code></pre>
      <p className="mt-3">
        Setting `cacheTTL: 0` disables caching entirely -- every permission check will hit the adapter.
        This is useful for testing but not recommended for production.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="cache-invalidation">
    <AccordionTrigger>How do I manually invalidate caches?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The engine provides several invalidation methods:
      <ul className="mt-2 list-disc pl-5">
        <li><code className="rounded bg-muted px-2 py-1">engine.invalidate()</code> -- clears all four caches</li>
        <li><code className="rounded bg-muted px-2 py-1">engine.invalidateSubject('user-1')</code> -- clears one subject's cached data</li>
        <li><code className="rounded bg-muted px-2 py-1">engine.invalidatePolicies()</code> -- clears the policy cache</li>
        <li><code className="rounded bg-muted px-2 py-1">engine.invalidateRoles()</code> -- clears the role and RBAC policy caches</li>
      </ul>
      <p className="mt-3">
        `engine.admin.*` methods do this automatically on mutation.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="performance">
    <AccordionTrigger>What is the performance like?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      With cached data, a typical evaluation takes under 1 ms. Data is loaded once and cached; each check
      only runs rule matching and condition evaluation. Use `engine.permissions()` for batch checks --
      it loads data once and evaluates all checks against it. `Decision.duration` (ms) lets you monitor
      evaluation times.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="batch-performance">
    <AccordionTrigger>Why is engine.permissions() faster than multiple engine.can() calls?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      `engine.permissions()` loads the subject, roles, and policies once, then evaluates all permission
      checks against the same loaded data. If you called `engine.can()` 10 times, even with caching
      you would have 10 separate async calls with cache lookups. `permissions()` does it in one call
      with one set of cache lookups. For UI permission maps with 5-20 checks, this makes a measurable
      difference.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="lru-eviction">
    <AccordionTrigger>What happens when the subject cache is full?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The subject cache uses an LRU (Least Recently Used) eviction policy. When the cache reaches
      `maxCacheSize` (default: 1000), the least recently accessed entry is evicted to make room
      for the new entry. This means frequently active users stay cached while inactive users are evicted.
      If you have more than 1000 concurrent active users, increase `maxCacheSize` accordingly.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="casl-comparison">
    <AccordionTrigger>How fast is duck-iam compared to CASL?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      CASL is roughly 2x faster on simple permission checks because it uses a pre-compiled ability
      index -- lookups are essentially hash-map hits. duck-iam runs a dynamic policy engine that
      evaluates conditions at runtime, which adds overhead per check. That said, duck-iam is faster
      than casbin, accesscontrol, and role-acl in our benchmarks. If raw check speed on simple
      RBAC-only rules is your top priority, CASL wins. If you need conditions, ABAC, scoped roles,
      or a unified RBAC+ABAC pipeline, duck-iam is a better fit. See{' '}
      <a href="/docs/benchmarks" className="underline">the benchmarks page</a> for detailed numbers.
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Adapters and Storage

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="what-adapter">
    <AccordionTrigger>What is an adapter?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      An adapter is the data layer interface that the engine uses to load and store authorization data
      (policies, roles, subject assignments). duck-iam provides four built-in adapters:
      <ul className="mt-2 list-disc pl-5">
        <li><strong>MemoryAdapter</strong> -- stores everything in memory. Great for testing and prototyping.</li>
        <li><strong>PrismaAdapter</strong> -- stores in a database via Prisma ORM.</li>
        <li><strong>DrizzleAdapter</strong> -- stores in a database via Drizzle ORM.</li>
        <li><strong>HttpAdapter</strong> -- fetches from a remote authorization service via HTTP.</li>
      </ul>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="memory-adapter-production">
    <AccordionTrigger>Can I use MemoryAdapter in production?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      You can, but be aware of the trade-offs. MemoryAdapter stores all data in process memory, so
      it is lost when the process restarts. It does not share state across multiple server instances.
      It is best for: testing, prototyping, single-instance applications with static role/policy
      definitions, or edge/serverless functions where you define everything in code.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="database-storage">
    <AccordionTrigger>How do I store policies in a database?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Use a database adapter:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`import { PrismaAdapter } from '@gentleduck/iam/adapters/prisma'
const adapter = new PrismaAdapter({ prisma: prismaClient })

// Or with Drizzle:
import { DrizzleAdapter } from '@gentleduck/iam/adapters/drizzle'
const adapter = new DrizzleAdapter({ db: drizzleInstance })`}</code></pre>
      <p className="mt-3">
        These adapters store policies, roles, and subject data in your database tables. When loading
        policies from external sources, always validate them with `validatePolicy()` before passing
        them to the engine.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="http-adapter">
    <AccordionTrigger>When should I use the HttpAdapter?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Use HttpAdapter when authorization data lives in a separate service. Configure the base URL and
      optional auth headers:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`import { HttpAdapter } from '@gentleduck/iam/adapters/http'

const adapter = new HttpAdapter({
  baseURL: 'https://auth-service.internal/api',
  headers: { 'Authorization': 'Bearer <service-token>' },
})`}</code></pre>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="custom-adapter">
    <AccordionTrigger>How do I write a custom adapter?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Implement the `Adapter` interface, which combines `PolicyStore`, `RoleStore`, and `SubjectStore`:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`interface Adapter extends PolicyStore, RoleStore, SubjectStore {}

// PolicyStore: listPolicies, getPolicy, savePolicy, deletePolicy
// RoleStore: listRoles, getRole, saveRole, deleteRole
// SubjectStore: getSubjectRoles, assignRole, revokeRole, getSubjectAttributes,
//              setSubjectAttributes, getSubjectScopedRoles (optional)`}</code></pre>
      <p className="mt-3">
        `getSubjectScopedRoles()` is optional -- if your adapter does not support multi-tenant scoping,
        omit it. Use the built-in `MemoryAdapter` source code as a reference implementation.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="adapter-peer-deps">
    <AccordionTrigger>Do database adapters add dependencies?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      No. Database adapters use peer dependencies -- `@prisma/client` for `PrismaAdapter`,
      `drizzle-orm` for `DrizzleAdapter`. Neither is bundled by duck-iam.
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Debugging and Troubleshooting

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="debug-denials">
    <AccordionTrigger>How do I debug permission denials?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Use `engine.explain()`:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const result = await engine.explain('user-1', 'update', {
  type: 'post',
  id: 'post-5',
  attributes: { ownerId: 'user-3' },
})
console.log(result.summary)`}</code></pre>
      <p className="mt-3">
        `result.summary` shows the final result, the subject's roles, which policies were evaluated,
        which rules matched or were skipped, and per-condition traces with expected vs actual values.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="explain-vs-check">
    <AccordionTrigger>What is the difference between explain() and check()?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      `check()` (and `can()`) run the evaluation and return a `Decision` (boolean + metadata).
      `explain()` runs the same evaluation but does NOT short-circuit -- it evaluates ALL policies
      and ALL rules even after a deny, so you get the complete picture. It returns an `ExplainResult`
      with full traces of every policy, rule, and condition. `explain()` still applies
      `beforeEvaluate`, but it does not trigger `afterEvaluate`, `onDeny`, or `onError`.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="targetmatch-deny-trace">
    <AccordionTrigger>Why does a PolicyTrace sometimes show targetMatch: false but result: deny?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      In explain output, a policy whose targets do not match still carries the engine's default
      effect as its `result`. The important signal is `targetMatch: false` plus an empty `rules`
      array, which tells you the policy's rules were never evaluated for that request.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="explain-missing-decision-rule">
    <AccordionTrigger>Why is ExplainResult.decision.rule sometimes empty even when decidingRuleId is present?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The explain trace records the deciding rule on each `PolicyTrace`, but the top-level
      `ExplainResult.decision.rule` is not currently populated on the explain path. Use
      `policyTrace.decidingRuleId`, `result.summary`, and `result.decision.policy` when you
      need the exact trace of which rule won.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="no-matching-rules">
    <AccordionTrigger>The summary says "No matching rules" -- what does that mean?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      It means no rules in any policy matched the request's action and resource combination. This
      usually means the subject does not have a role that grants that permission. Check:
      <ul className="mt-2 list-disc pl-5">
        <li>`result.subject.roles` -- does the subject have the expected roles?</li>
        <li>Is the action spelled correctly? ("read" vs "Read")</li>
        <li>Is the resource type spelled correctly? ("post" vs "posts")</li>
        <li>Did you pass the scope parameter if the roles are scoped?</li>
      </ul>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="condition-failed">
    <AccordionTrigger>A condition is failing but I think it should pass. How do I debug it?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Look at the condition trace in the `ExplainResult`. Each `ConditionLeafTrace` shows:
      <ul className="mt-2 list-disc pl-5">
        <li><strong>field</strong> -- the path being checked (e.g., `resource.attributes.ownerId`)</li>
        <li><strong>expected</strong> -- the value from the condition definition</li>
        <li><strong>actual</strong> -- the value resolved from the request at runtime</li>
        <li><strong>result</strong> -- true or false</li>
      </ul>
      <p className="mt-3">
        Common problems: the field path is wrong (`author` vs `ownerId`), the actual value is `null`
        because the attribute was not passed, or a `$` variable was not resolved because the request
        context is missing that field.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="scoped-roles-not-applied">
    <AccordionTrigger>My scoped roles are not being applied. What is wrong?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Check with `explain()`:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const result = await engine.explain('user-1', 'manage',
  { type: 'dashboard', attributes: {} },
  undefined,  // environment
  'org-1',    // scope
)
console.log('Base roles:', result.subject.roles)
console.log('Scoped roles added:', result.subject.scopedRolesApplied)`}</code></pre>
      <p className="mt-3">
        If `scopedRolesApplied` is empty: the adapter does not implement `getSubjectScopedRoles()`,
        there are no scoped assignments for this subject/scope, or the scope parameter was not passed.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="unexpected-allow">
    <AccordionTrigger>Why was my request allowed when it should be denied?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Use `explain()` and look at which policies allowed it:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`for (const pt of result.policies) {
  if (pt.result === 'allow') {
    console.log('Policy "' + pt.policyId + '" allowed:')
    for (const rule of pt.rules.filter(r => r.matched && r.effect === 'allow')) {
      console.log('  Rule "' + rule.ruleId + '" matched')
    }
  }
}`}</code></pre>
      <p className="mt-3">
        Common causes: a wildcard permission (`grant('*', '*')`), a policy with
        `allow-overrides` that is too broad, or a deny policy whose target filter does not match the request.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="validation-errors">
    <AccordionTrigger>Common mistakes caught by validatePolicy()?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">

## Security and Edge Cases

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="fail-closed">
    <AccordionTrigger>What does "fail closed" mean?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      On any error, the system defaults to deny:
      <ul className="mt-2 list-disc pl-5">
        <li>If no rules match, the result is deny (via `defaultEffect: 'deny'`)</li>
        <li>If an adapter call throws an error, the result is deny</li>
        <li>If a condition evaluation fails, that condition returns false</li>
        <li>If a `beforeEvaluate` hook throws, the result is deny</li>
        <li>If the subject is not found, the result is deny</li>
      </ul>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="prototype-pollution">
    <AccordionTrigger>Is duck-iam protected against prototype pollution?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. The field resolver blocks `__proto__`, `constructor`, and `prototype` in all dot-paths.
      A field like `resource.attributes.__proto__.polluted` fails safely without resolving.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="allowed-blocked-lists">
    <AccordionTrigger>Why does field resolution have an allowed list and a blocked list?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      `resolve()` uses two safety mechanisms:
      <ul className="mt-2 list-disc pl-5">
        <li><strong>Allowed list</strong> (`subject`, `resource`, `environment`) -- only these top-level
          keys can be traversed. `action` and `scope` are handled as scalars before traversal starts.</li>
        <li><strong>Blocked list</strong> (`__proto__`, `constructor`, `prototype`) -- blocked at every
          traversal level to prevent prototype chain access.</li>
      </ul>
      <p className="mt-3">
        This is especially important when policies come from untrusted sources (databases, APIs). With
        <code className="rounded bg-muted px-2 py-1">createAccessConfig()</code>, <code className="rounded bg-muted px-2 py-1">DollarPaths&lt;TContext&gt;</code> enforces
        the same constraints at compile time -- autocomplete only shows valid paths.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="defense-in-depth">
    <AccordionTrigger>What is defense-in-depth in the context of duck-iam?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Cross-policy AND-combination: every policy is an independent security gate. RBAC must allow AND
      every ABAC policy must allow. A deny from any single policy means overall deny. New restriction
      policies add protection without risk of existing policies overriding them.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="client-only-security">
    <AccordionTrigger>Can I rely on client-side permission checks for security?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      No. Client-side checks are for UI rendering only (show/hide buttons, enable/disable actions).
      Client-side checks are for rendering only. A user can bypass them by modifying JavaScript or
      calling the API directly. Always enforce permissions on the server.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="no-roles-assigned">
    <AccordionTrigger>What happens when a subject has no roles assigned?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The `__rbac__` policy has no matching rules and returns deny. Unless an ABAC policy explicitly
      allows roleless subjects, every check denies. No roles means no access.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="subject-not-found">
    <AccordionTrigger>What happens when the subject does not exist in the adapter?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The subject is treated as having no roles and no attributes -- zero RBAC permissions, no subject
      attributes for conditions. Result is deny unless an ABAC policy explicitly allows access without
      subject-based conditions.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="empty-policy-rules">
    <AccordionTrigger>What happens if a policy has no rules?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Returns the default effect. With the default configuration (`defaultEffect: 'deny'`), an empty
      policy always denies regardless of combining algorithm.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="max-conditions-depth">
    <AccordionTrigger>Can deeply nested conditions cause stack overflows?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      No. Condition groups are limited to 10 levels deep. Beyond that, evaluation stops and returns
      false (fail closed), preventing both accidental complexity and DoS via deeply nested trees.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="regex-safety">
    <AccordionTrigger>Can the `matches` operator cause ReDoS attacks?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The `matches` operator creates a `RegExp` from the condition value. If untrusted users can
      define policy conditions (e.g., via an admin dashboard), they could craft a malicious regex
      pattern that causes catastrophic backtracking. To mitigate this: validate regex patterns
      before saving policies, limit who can create conditions with `matches`, or use simpler
      operators like `starts_with`, `ends_with`, or `eq` when possible.
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Type Safety

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="typed-vs-untyped">
    <AccordionTrigger>What is the difference between typed and untyped mode?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      <strong>Untyped (direct imports):</strong> Import `defineRole`, `policy`, `Engine` directly from
      `@gentleduck/iam`. Actions and resources accept any string. Quick and flexible, but typos are not caught
      at compile time.
      <br /><br />
      <strong>Typed (createAccessConfig):</strong> Call `createAccessConfig()` with `as const` arrays.
      Every builder method is constrained to your declared schema. Misspell an action and TypeScript
      catches it before your code runs.
      <br /><br />
      Use the typed approach in production. It prevents silent failures from typos in action or resource names.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="as-const">
    <AccordionTrigger>Why is `as const` required for createAccessConfig()?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Without `as const`, TypeScript widens arrays to `string[]`:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`// Without as const:
const actions = ['create', 'read']  // type: string[]

// With as const:
const actions = ['create', 'read'] as const  // type: readonly ['create', 'read']`}</code></pre>
      <p className="mt-3">
        `createAccessConfig()` needs literal types to constrain builder methods. With `string[]`,
        typos like `'craete'` are indistinguishable from valid values.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="typed-checks">
    <AccordionTrigger>What does access.checks() do?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      `access.checks()` is a compile-time-only utility -- it returns the input unchanged but constrains
      action/resource types. Use it with `engine.permissions()` for typed batch checks:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const uiChecks = access.checks([
  { action: 'create', resource: 'post' },
  { action: 'manage', resource: 'dashboard' },
  // { action: 'approve', resource: 'post' }  // ERROR: 'approve' is not valid
])

const perms = await engine.permissions('user-1', uiChecks)`}</code></pre>
      <p className="mt-3">
        At runtime, `access.checks()` is a no-op. Its only purpose is compile-time validation.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="typed-engine">
    <AccordionTrigger>Is the engine typed when created via access.createEngine()?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. When you use `access.createEngine()` instead of `new Engine()`, the engine's `can()`,
      `check()`, `explain()`, and `permissions()` methods are all constrained to your schema's
      actions and resources. Passing an invalid action or resource is a compile error.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="omit-scopes">
    <AccordionTrigger>Can I omit scopes from createAccessConfig()?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. Scopes are optional in the config:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const access = createAccessConfig({
  actions: ['create', 'read', 'update', 'delete'] as const,
  resources: ['post', 'comment'] as const,
  // scopes omitted -- scope parameters accept 'string' without constraint
})`}</code></pre>
      <p className="mt-3">
        If you omit scopes, scope parameters in the engine will accept any string. You lose
        type-checking for scopes but everything else is still constrained.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="config-vs-standalone">
    <AccordionTrigger>What is the difference between createAccessConfig() builders and standalone defineRole() / defineRule()?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The difference is compile-time type safety. Both produce identical output at runtime.
      <br /><br />
      <strong>Standalone builders</strong> default their generics to `string`, so they accept anything:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`import { defineRole } from '@gentleduck/iam'

const viewer = defineRole('viewer')
  .grant('read', 'post')
  .grant('deletee', 'postt')  // no error -- both are valid strings
  .build()`}</code></pre>
      <p className="mt-3">
        TypeScript sees `'deletee'` as a valid `string`. The role ends up with a permission that
        never matches -- a silent bug you may not discover until production.
      </p>
      <strong>Config-bound builders</strong> lock generics to your declared schema:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`import { createAccessConfig } from '@gentleduck/iam'

const access = createAccessConfig({
  actions: ['create', 'read', 'update', 'delete'] as const,
  resources: ['post', 'user'] as const,
})

const viewer = access
  .defineRole('viewer')
  .grant('read', 'post')      // OK
  .grant('deletee', 'postt')  // compile error on both arguments
  .build()`}</code></pre>
      <p className="mt-3">Applies to every builder on the config object:</p>
      <ul className="mt-2 list-disc pl-5">
        <li><code className="rounded bg-muted px-2 py-1">access.defineRole()</code> -- constrained <code className="rounded bg-muted px-2 py-1">.grant()</code> and <code className="rounded bg-muted px-2 py-1">.inherits()</code></li>
        <li><code className="rounded bg-muted px-2 py-1">access.defineRule()</code> -- constrained <code className="rounded bg-muted px-2 py-1">.on()</code> and <code className="rounded bg-muted px-2 py-1">.of()</code></li>
        <li><code className="rounded bg-muted px-2 py-1">access.policy()</code> -- constrained rule actions and resources</li>
        <li><code className="rounded bg-muted px-2 py-1">access.when()</code> -- constrained condition builder</li>
        <li><code className="rounded bg-muted px-2 py-1">access.checks()</code> -- constrained permission check definitions</li>
        <li><code className="rounded bg-muted px-2 py-1">access.createEngine()</code> -- engine typed with your schema, so <code className="rounded bg-muted px-2 py-1">engine.can()</code> is also constrained</li>
      </ul>
      <p className="mt-3">
        Both approaches produce identical <code className="rounded bg-muted px-2 py-1">Role</code> and
        <code className="rounded bg-muted px-2 py-1">Policy</code> objects at runtime. The difference is compile-time safety.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="export-types-from-config">
    <AccordionTrigger>How do I share my action and resource types across the codebase?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Extract type aliases from your config and export them:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`// shared/access.ts
export const access = createAccessConfig({
  actions: ['create', 'read', 'update', 'delete'] as const,
  resources: ['post', 'user'] as const,
})

export type AppAction = (typeof access.actions)[number]
// => 'create' | 'read' | 'update' | 'delete'

export type AppResource = (typeof access.resources)[number]
// => 'post' | 'user'`}</code></pre>
      <p className="mt-3">Use these types in guards, decorators, components, and API types:</p>
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`// api/access/guard.ts
import type { Engine } from '@gentleduck/iam'
import type { AppAction, AppResource } from '@shared/access'

constructor(@Inject(ACCESS_ENGINE_TOKEN) engine: Engine<AppAction, AppResource>) { ... }

// api/access/authorize.ts
import { createTypedAuthorize } from '@gentleduck/iam/server/nest'
import type { AppAction, AppResource } from '@shared/access'

export const Authorize = createTypedAuthorize<AppAction, AppResource>()`}</code></pre>
      <p className="mt-3">
        Single source of truth: add an action or resource to the config and TypeScript surfaces every
        call site that needs updating.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="can-mix-approaches">
    <AccordionTrigger>Can I mix config-bound and standalone builders?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes -- both produce identical <code className="rounded bg-muted px-2 py-1">Role</code> and
      <code className="rounded bg-muted px-2 py-1">Policy</code> objects at runtime. But mixing them is pointless: a single
      standalone `defineRole()` with a typo defeats the type safety of `createAccessConfig()`. Pick one.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="dollar-paths-autocomplete">
    <AccordionTrigger>How do I get autocomplete for $-variable references?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      When you use <code className="rounded bg-muted px-2 py-1">createAccessConfig()</code> with a typed context,
      all <code className="rounded bg-muted px-2 py-1">$</code>-references get full IDE autocomplete. The
      <code className="rounded bg-muted px-2 py-1">DollarPaths&lt;TContext&gt;</code> type generates
      <code className="rounded bg-muted px-2 py-1">$</code>-prefixed versions of all valid dot-paths from your context type.
      <p className="mt-3">
        In value positions of <code className="rounded bg-muted px-2 py-1">.check()</code>,
        <code className="rounded bg-muted px-2 py-1">.eq()</code>, <code className="rounded bg-muted px-2 py-1">.neq()</code>,
        <code className="rounded bg-muted px-2 py-1">.attr()</code>, <code className="rounded bg-muted px-2 py-1">.resourceAttr()</code>,
        and <code className="rounded bg-muted px-2 py-1">.env()</code>, typing
        <code className="rounded bg-muted px-2 py-1">'$'</code> triggers suggestions like:
      </p>
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`'$subject.id'
'$subject.roles'
'$resource.attributes.ownerId'
'$environment.hour'`}</code></pre>
      <p className="mt-3">
        No <code className="rounded bg-muted px-2 py-1">as string</code> cast is needed. The
        <code className="rounded bg-muted px-2 py-1">& {"{}"}</code> trick is used internally to prevent TypeScript
        from collapsing <code className="rounded bg-muted px-2 py-1">string | '$subject.id'</code> into just
        <code className="rounded bg-muted px-2 py-1">string</code>, which would eliminate autocomplete.
      </p>
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Common Patterns

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="owner-access">
    <AccordionTrigger>How do I implement owner-only access?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Create a policy that denies non-owners:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const ownerPolicy = policy('owner-only')
  .algorithm('deny-overrides')
  .rule('deny-non-owner', r => r
    .deny()
    .on('update', 'delete')
    .of('post')
    .priority(100)
    .when(w => w
      .check('resource.attributes.ownerId', 'neq', '$subject.id')
      .not(w => w.role('admin'))  // admins are exempt
    )
  )
  .build()`}</code></pre>
      <p className="mt-3">
        `$subject.id` resolves at runtime. Combined with `neq` and `deny`, this blocks non-owners.
        `.not()` exempts admins. Pass `ownerId` in resource attributes.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="time-based-access">
    <AccordionTrigger>How do I implement time-based access restrictions?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Pass time information in the environment and check it in conditions:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`// When checking:
await engine.can('user-1', 'deploy', resource, {
  dayOfWeek: new Date().toLocaleDateString('en', { weekday: 'long' }),
  hour: new Date().getHours(),
})

// In a policy rule:
.when(w => w
  .check('environment.dayOfWeek', 'nin', ['Saturday', 'Sunday'])
  .check('environment.hour', 'gte', 9)
  .check('environment.hour', 'lt', 17)
)`}</code></pre>
      <p className="mt-3">
        This restricts deployments to weekdays between 9 AM and 5 PM.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="admin-bypass">
    <AccordionTrigger>How do I let admins bypass all restrictions?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Option 1: Give the admin role wildcard permissions:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const admin = defineRole('admin').grant('*', '*').build()`}</code></pre>
      Option 2: Exclude admins from deny conditions using `.not()`:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`.when(w => w
  .check('resource.attributes.ownerId', 'neq', '$subject.id')
  .not(w => w.role('admin'))  // admin is exempt from this deny
)`}</code></pre>
      <p className="mt-3">
        Option 1: full RBAC bypass. Option 2: targeted ABAC exemption. Combine both for maximum flexibility.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="resource-status">
    <AccordionTrigger>How do I restrict actions based on resource status?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Pass the status as a resource attribute and check it in conditions:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`// When checking:
await engine.can('user-1', 'update', {
  type: 'post',
  id: 'post-1',
  attributes: { status: 'locked', ownerId: 'user-1' },
})

// In a policy:
.rule('deny-locked-edit', r => r
  .deny()
  .on('update', 'delete')
  .of('post')
  .when(w => w.check('resource.attributes.status', 'eq', 'locked'))
)`}</code></pre>
      <p className="mt-3">
        This denies editing or deleting locked posts regardless of who the user is.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="ip-filtering">
    <AccordionTrigger>How do I restrict access by IP address?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Pass the IP in the environment and check it in conditions:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`await engine.can('user-1', 'manage', resource, {
  ip: req.ip,
})

// Policy rule:
.rule('office-only', r => r
  .deny()
  .on('manage')
  .of('settings')
  .when(w => w.check('environment.ip', 'not_contains', '10.0.'))
)`}</code></pre>
      <p className="mt-3">
        For more sophisticated IP matching, use the `matches` operator with a regex pattern.
      </p>
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="feature-flags">
    <AccordionTrigger>Can I use duck-iam for feature flags?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes, though it is not the primary use case. Model features as resources:
      <pre className="mt-3 rounded bg-muted p-3 text-sm overflow-x-auto"><code>{`const betaTester = defineRole('beta-tester')
  .grant('access', 'feature-dark-mode')
  .grant('access', 'feature-ai-assistant')
  .build()

await engine.can('user-1', 'access', { type: 'feature-dark-mode', attributes: {} })`}</code></pre>
      <p className="mt-3">
        Works for role-based flags. For percentage rollouts or A/B testing, use a dedicated feature flag service.
      </p>
    </AccordionContent>
  </AccordionItem>
</Accordion>

## Architecture and Adoption

<Accordion type="multiple" collapsible className="w-full">
  <AccordionItem value="memory-first-migrate-later">
    <AccordionTrigger>Can I start with MemoryAdapter and migrate later?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Yes. That is a common adoption path. The engine API stays the same, so migration is mostly about moving your roles,
      policies, and assignments into persistent storage and swapping the adapter.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="permission-maps-where">
    <AccordionTrigger>Where should permission maps be generated for frontend apps?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      On the server, from the same engine and adapter setup that enforces your real authorization rules. The client should
      usually receive the resulting <code className="rounded bg-muted px-2 py-1">PermissionMap</code>, not the raw policy system.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="serialized-permission-keys">
    <AccordionTrigger>Why does duck-iam use serialized permission keys like scope:action:resource:id?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      That shared key format makes permission maps portable across the engine, server integrations, client libraries, and
      caches. Scope and resource instance IDs are part of the permission identity, so they must be encoded into the key.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="missing-user-id">
    <AccordionTrigger>What happens if no user ID is present in a server integration?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      The default behavior is fail-closed. Server integrations treat a missing identity as unauthorized and return a
      <code className="rounded bg-muted px-2 py-1">401</code> response unless you override that behavior.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="tenant-scope-integrations">
    <AccordionTrigger>How should I handle tenant scope in framework integrations?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Pass a <code className="rounded bg-muted px-2 py-1">getScope</code> function or fixed scope where the integration supports it,
      and generate permission maps per scope when hydrating clients. Treat scope as request data, not just configuration.
    </AccordionContent>
  </AccordionItem>

  <AccordionItem value="production-without-admin-ui">
    <AccordionTrigger>What is the safest production path if I do not want an admin UI yet?</AccordionTrigger>
    <AccordionContent className="text-muted-foreground">
      Keep policy and role definitions in migrations, seed scripts, or reviewed startup code, persist them with Prisma or
      Drizzle, validate them at startup, and expose only the minimal assignment flows your application actually needs.
    </AccordionContent>
  </AccordionItem>
</Accordion>
